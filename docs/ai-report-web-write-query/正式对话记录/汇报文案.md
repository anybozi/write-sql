# 第1页｜项目定位

## AI自然语言取数能力底座

### 一句话定位

将业务自然语言需求，转化为准确、可解释、可校验的 CDAP Hive SQL。

### 核心观点

自然语言取数的难点，不是让 AI “会写 SQL”，而是让 AI 知道：

* 业务人员真正想查什么
* 应该选择哪张主表
* 应该采用什么业务口径
* 字段不足时应该补哪张表
* 生成结果需要检查哪些风险

本项目将数据支撑人员长期积累的取数经验，沉淀为一套可运行、可复用、可持续完善的 AI 技能包，让 AI 按照真实取数流程逐步生成 SQL，而不是仅凭字段名称直接猜表、猜口径。

### 三项核心价值

**准确生成**
按照业务口径、表结构和字段规则生成 SQL。

**经验沉淀**
将个人经验转化为可复用、可持续补充的技能资产。

**能力复用**
为后续业务门户、智能问数、机器人等应用提供自然语言转 SQL 的基础能力。

### 页面总结语

**先把自然语言转 SQL 做准，再为后续业务自然语言取数提供能力支撑。**

# 第2页｜背景与痛点

## 数据需求持续增长，传统取数高度依赖个人经验

### 业务背景

经营分析、指标跟踪、营销支撑和专项核查等场景，对数据取数的需求越来越频繁。

业务人员通常使用自然语言提出问题，而数据库只能执行结构化 SQL。支撑人员需要在两者之间完成需求理解、口径转换、选表、字段匹配、SQL 编写和结果检查。

### 当前取数面临四个主要问题

**01 需求收口难**
一句业务需求背后，往往还需要确认查什么、查谁、按什么时间统计、输出什么字段、是否有状态或区域限制。

**02 口径判断难**
“入网、到达、在网、出账、拆机、收入、积分”等业务词，对应不同事实、时间字段和过滤条件。

**03 选表补字段难**
CDAP 表数量多、结构复杂。主表怎么选、字段从哪里取、什么时候补维表或事实表，主要依赖熟手经验。

**04 重复工作量大**
大量需求虽然业务内容不同，但处理步骤高度相似，每次仍需人工重复理解需求、查表、确认口径、写 SQL、检查结果。

### 模式转变

**传统方式：**
业务人员描述需求，支撑人员依靠个人经验从零完成 SQL 编写。

**技能方式：**
业务人员描述需求，AI 按照沉淀的取数经验生成 SQL，支撑人员重点进行确认和审核。

### 页面总结语

**从“个人经验驱动取数”，逐步转向“知识资产与 AI 协同取数”。**

# 第3页｜能力运行流程与技能架构

## 不是让 AI 直接猜 SQL，而是让 AI 按取数经验逐步生成

### 一、能力运行流程

**01 需求拆解与澄清**
AI 先识别查什么、查谁、按什么时间算、怎么输出结果、有什么业务限制。只追问会改变 SQL 结果的关键问题。

**02 主表选择**
根据“业务对象、动作或状态、输出粒度”判断业务事实发生在哪张表。主表由业务事实决定，而不是由字段名称相似度决定。

**03 业务口径确认**
判断需求是否命中标准指标，并确认业务口径、时间字段、状态或动作条件、产品及客户范围。

**04 字段映射与补表**
先判断主表是否已有所需字段。主表字段不足时，再根据补表规则选择维表或辅助事实表，并检查关联键、数据粒度和行数放大风险。

**05 SQL 生成**
根据已经确认的主表、口径、字段和过滤条件，选择单 SELECT 或 CTAS 流水线，生成完整 Hive SQL。

**06 规则校验与结果自检**
SQL 生成后同步检查主表、字段来源、时间分区、JOIN 粒度、状态码值、过程表行数、样例量级和关键字段质量。

### 二、技能资产架构

```text
write-query/
├── SKILL.md
│   └── 技能入口、路径判断与整体运行流程
│
└── references/
    ├── TABLE_INDEX.md
    │   └── CDAP 表资产索引
    │
    ├── ROUTING.md
    │   └── 自然语言需求到主事实表的路由规则
    │
    ├── METRIC_INDEX.md
    │   └── 标准指标索引与指标口径入口
    │
    ├── FIELD_BACKFILL.md
    │   └── 字段缺口、补表路径与 JOIN 风险规则
    │
    ├── RULES.md
    │   └── SQL 生成规范、规则校验和自检要求
    │
    ├── tables/
    │   └── CDAP 业务表文档
    │
    ├── metrics/
    │   └── 标准指标技术口径
    │
    ├── verified-cases/
    │   └── 已验证的复杂取数案例
    │
    └── D_experience/dictionaries/
        └── 状态、动作和业务码值字典
```

### 运行原则

技能启动后不会一次性加载全部资料，而是根据当前步骤按需读取：

* 选主表时读取表索引和路由
* 确认指标时读取指标口径
* 映射字段时读取目标表文档
* 出现字段缺口时读取补表规则
* 生成 SQL 时读取生成与校验规则

### 页面总结语

**上层是标准化的取数流程，下层是支撑流程运行的业务知识资产。**

# 第4页｜真实案例验证

## 从一句业务需求，到一套可确认的取数 SQL

### 用户原始需求

> 请教下，能帮忙取一下近三个月，机主年龄大于等于65岁的所有订单不？

这句话看似简单，但直接生成 SQL 仍存在多个会影响结果的关键问题。

### 第一步：AI 只澄清三个关键问题

**01 近三个月按什么时间计算？**
是订单受理时间 `act_date`，还是竣工 / 状态时间 `subs_stat_date`。

**02 年龄按什么时间点计算？**
是当前最新资料月年龄，还是订单发生当月的年龄快照。

**03 输出明细还是汇总？**
是输出全部订单明细，还是按月、分局、营服汇总。

### 用户确认结果

* 时间：按受理时间，查询 202604—202606
* 年龄：按当前最新资料月计算
* 结果：输出订单明细
* 字段：订单 ID、订单编码、号码、客户、年龄、受理时间、网点信息、揽装信息等

### 第二步：AI 形成取数方案

**主表选择：040 全业务号码订单表**
原因：用户查询的是“所有订单明细”，订单事实应从订单表出发。

同时读取：

* 当前订单表：`zone_gz_yz.dwm_yz_rpt_comm_ba_subs_final`
* 历史月订单表：`zone_gz_yz.dwm_yz_rpt_comm_ba_subs_mon_final`

**时间口径：**

```text
2026-04-01 <= act_date < 2026-07-01
```

`par_month_id` 只用于读取归档分区，不代替受理时间条件。

**年龄补表：069 全业务资料表当前最新资料月**

```text
040.serv_id = 069.serv_id
```

使用身份证出生年份计算年龄，但不输出身份证原文。

**“所有订单”口径：**

* 不限制只取竣工订单
* 不默认排除撤单、作废
* 保留订单全流程状态

### 第三步：AI 主动识别风险

**风险一：年龄时间点**
年龄按照当前资料月计算，不代表订单发生时的年龄。

**风险二：订单重复**
当前表和历史月表可能同时存在同一订单，因此按 `subs_id` 去重，并优先保留当前表记录。

**风险三：补表放大**
069 当前月同一个 `serv_id` 理论上应唯一，但仍需要检查 JOIN 前后行数，防止结果放大。

### 第四步：生成完整 SQL 与自检方案

本次案例最终形成：

* 3 个关键澄清问题
* 3 个生产数据来源
* 4 段 CTAS 脚本
* 3 项风险提示
* 7 条自检 SQL

自检内容覆盖：

* 订单行数、订单数、服务数
* 年龄资料快照月份
* 各受理月份数据量
* 订单状态和状态原因分布
* 合并订单池是否重复
* 069 补表是否放大
* 最终结果样例

### 传统方式与技能方式对比

| 传统方式       | 技能方式               |
| ---------- | ------------------ |
| 人工判断需求缺口   | AI 主动识别关键缺口        |
| 人工查找订单主表   | AI 按业务事实选择 040 订单表 |
| 人工判断年龄来源   | AI 补 069 当前资料月     |
| 人工处理当前表和月表 | AI 合并并按订单 ID 去重    |
| 人工从零编写脚本   | AI 生成 CTAS 脚本      |
| 人工自行检查     | AI 同步生成自检 SQL      |

### 页面总结语

**AI 不只是输出 SQL，更完成了需求收口、选表、口径、补表、风险识别和结果自检。**

# 第5页｜技能资产与能力沉淀

## 将个人经验沉淀为可持续积累的 AI 技能资产

### 核心观点

自然语言取数的准确性，不只依赖大模型，更依赖模型背后的业务知识资产。

本项目将日常取数中积累的选表、口径、字段、补表和检查经验进行结构化沉淀，使这些经验能够被 AI 按需调用、持续完善和重复使用。

### 当前核心资产

| 技能资产       | 主要解决的问题                         |
| ---------- | ------------------------------- |
| CDAP 表索引   | 帮助 AI 定位业务事实表、生产表名、粒度、分区和适用范围   |
| 标准指标索引     | 帮助 AI 按照统一业务口径生成 SQL            |
| 主表路由规则     | 解决自然语言需求应从哪张主事实表出发              |
| 字段补表规则     | 解决主表字段不足时补哪张表、如何关联              |
| SQL 规则与自检项 | 检查主表、字段、口径、时间、JOIN、码值和可执行性      |
| 专项场景规则     | 针对入网、收入、CRM、国漫、IMSI、地址等场景检查关键风险 |
| 已验证案例      | 为复杂、多表和历史回溯需求提供成熟生成路径           |
| 业务码值字典     | 将状态、动作和原因转换为可执行技术条件             |

### 标准化数据字典的作用

前期标准化数据字典沉淀了部分指标口径和维度字段成果，例如：

* 指标口径：宽带入网、移动入网、宽带有效率、拆机率等
* 维度字段：月份、分局、营服、网点、客户、产品、人员等

这些成果为自然语言取数提供了重要的标准化基础，但本技能并不止于数据字典，还进一步融合了主表路由、字段补表、SQL 规则、码值字典和真实案例。

### 从“文件数量”进一步形成“能力覆盖”

技能资产不仅回答“有什么”，还要回答：

* 需求应该从哪张表出发
* 普通需求和标准指标如何区分
* 动作事实与存量事实如何区分
* 日表和月表如何选择
* 主表字段不足时如何补表
* 一对多 JOIN 如何处理
* 复杂需求如何拆成过程表
* SQL 生成后如何验证结果

### 能力沉淀闭环

```text
真实取数需求
      ↓
发现新的业务规则或风险
      ↓
通过 SQL 结果进行验证
      ↓
回填表、指标、路由、补表或案例资产
      ↓
后续相似需求直接复用
```

### 页面总结语

**每完成一次真实需求，不只是生成一条 SQL，也是在持续丰富自然语言取数能力底座。**

# 第6页｜下一步规划

## 持续丰富能力底座，逐步提升生成准确性和场景覆盖

### 方向一：持续沉淀真实取数经验

围绕内部真实需求，持续补充和优化：

* CDAP 表文档及字段说明
* 标准指标与业务口径
* 主表路由规则
* 字段补表路径
* 状态、动作和业务码值
* 复杂需求验证案例
* SQL 规则和自检方式

让技能从“能够生成”，逐步发展到：

```text
覆盖更广
生成更准
解释更清楚
使用更稳定
```

### 方向二：建立试用、反馈和迭代机制

先在内部数据支撑人员中开展小范围试用。

```text
小范围试用
→ 收集真实需求
→ 跟踪生成效果
→ 发现问题
→ 修正规则
→ 扩大使用范围
```

### 建议重点跟踪的指标

**使用覆盖**

* 日常取数需求总量
* 技能辅助处理需求数量
* 技能需求覆盖率
* 活跃使用人数

**生成质量**

* SQL 初稿可用率
* SQL 准确率
* 一次通过率
* 人工修改率
* 主要问题类型分布

**效率提升**

* 传统处理平均时长
* 技能辅助后平均时长
* 单个需求节省时间
* 整体效率提升比例

**能力成长**

* 新增表文档数量
* 新增指标数量
* 新增路由及补表规则数量
* 新增验证案例数量
* 问题修正与知识回填数量

### 方向三：建立统一的知识回填规则

| 新经验类型         | 回填位置   |
| ------------- | ------ |
| 新业务术语和主表选择经验  | 主表路由   |
| 新标准指标和技术口径    | 指标库    |
| 新字段来源和补表路径    | 字段补表规则 |
| 新状态、动作和原因码值   | 码值字典   |
| 新 SQL 风险和通用写法 | SQL 规则 |
| 新复杂多表需求       | 已验证案例  |

形成：

**边使用、边验证、边沉淀、边提升。**

### 方向四：支撑后续应用扩展

当自然语言转 SQL 能力逐步成熟后，可作为统一能力组件，为后续应用按需提供支持，例如：

* 业务门户自然语言取数
* 企业微信机器人智能问数
* 数据服务平台智能查询
* 经营指标智能问答
* 业务数据需求自动生成

后续应用可以复用同一套主表路由、指标口径、字段补表、SQL 生成、规则校验与结果自检能力。

### 三阶段目标

**第一阶段：能用**
覆盖高频常规取数需求，能够生成结构完整的 SQL 初稿。

**第二阶段：好用**
通过持续试用和规则优化，提高 SQL 准确率、一次通过率，减少人工修改。

**第三阶段：复用**
将能力沉淀为稳定组件，为后续不同业务入口提供自然语言取数能力支持。

### 最终收尾

目前，AI 自然语言取数能力已形成雏形并投入实际使用。能力建设既吸收了标准化数据字典中的指标口径和维度字段成果，也融合了主表路由、字段补表、SQL 规则和真实案例。下一步将坚持在使用中验证、在验证中沉淀，持续提升生成准确性和场景覆盖能力。
